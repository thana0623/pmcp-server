报告 bug 并结构化修复。诊断 → 修复 → 提示重测 → 轮次管控。

## 使用方式

手动测试后报告 bug：
```
/test <bug 描述>
```

示例：
```
/test 发票模块接口返回 500
/test 前端列表页点击编辑没反应
/test 提交表单后数据没保存
```

## 步骤

### Step 1: 诊断

**禁止跳过诊断直接修复。**

1. 根据 bug 描述，用 codegraph 或 grep 定位相关文件和函数
2. 读取相关代码，分析根因
3. 输出诊断结果：

```
## 🔍 诊断

- 根因: <一句话>
- 文件: <相关文件>
- 影响: <影响范围>
```

如果不确定根因，问用户补充信息，不要猜。

### Step 2: 修复

按诊断结果直接修复代码。修复后告诉用户改了什么。

### Step 3: 记录轮次

更新 HOT_STATE.md，在"最近讨论"中记录：
- 修了什么 bug
- 第几轮（同一 bug 累计）

### Step 4: 提示重测

```
已修复。请重新测试验证。
```

### Step 5: 轮次管控

同一个 bug 修了 5 轮仍未解决：

```
## 🛑 已修 5 轮，建议开新会话

上下文可能已膨胀，修复质量下降。

建议：
1. /end 归档当前进度
2. 开新会话 /start 继续
3. 新会话里重新描述 bug，换个角度分析
```

## 注意

- 诊断优先，不要瞎试
- 每次只修一个 bug，不要同时改多个地方
- 修完必须让用户测试，不要自己"假设修好了"
- 5 轮是硬限制，到了就 /end
