测试阶段的结构化修复流程：诊断模块 → 修复 plan → 执行修复 → 轮次管控。

## 使用方式

用户在手动测试项目后报告 bug：
```
/test <bug 描述>
```

示例：
```
/test 发票模块的接口返回 500 错误
/test 前端列表页点击编辑按钮没反应
/test 提交表单后数据没保存到数据库
```

## 步骤

### Step 1: 记录 bug 到 state.md

读取 `.github/prompts/state.md`，在"发现的问题"中追加本次 bug：

```
- [轮次 N] <bug 描述>（<时间>）
```

轮次 N 的计算：读取"发现的问题"中同一功能/模块的已有条目数 + 1。

### Step 2: 模块诊断

**必须先诊断，再动手。禁止跳过。**

1. 根据 bug 描述，用 `codegraph_search` 或 `grep` 定位相关模块和文件
2. 用 `codegraph_context` 获取模块上下文
3. 用 `module-read <模块名>` 读取模块历史（如有）
4. 分析 bug 的根因：是哪个模块、哪个函数、什么逻辑出了问题

输出诊断结果：

```
## 🔍 诊断结果

- **问题模块**: <模块名>
- **根因**: <简要描述>
- **影响范围**: <哪些功能受影响>
- **相关文件**: <文件列表>
```

### Step 3: 生成修复 plan

将修复方案写入 `.github/prompts/plans/fix-<简短描述>.md`：

```markdown
# Fix: <bug 描述>

## 诊断
- 模块: <模块名>
- 根因: <根因>

## 修复方案
1. <步骤 1>
2. <步骤 2>

## 影响范围
- <受影响的文件/功能>

## 验证方式
- <如何验证修复成功>
```

展示给用户确认。

### Step 4: 执行修复

用户确认后，按 plan 执行修复。

修复完成后：
1. 用 `module-log` 记录模块修改
2. 更新 `.github/prompts/state.md` 的"进度"

### Step 5: 提示重新测试

```
## ✅ 修复完成

- Bug: <bug 描述>
- 模块: <模块名>
- 修改: <改了什么>

请重新测试验证。如果问题仍存在，再次 /test 报告。
```

### Step 6: 轮次管控

检查"发现的问题"中同一模块的轮次：

- **1-2 轮**: 正常迭代，继续
- **3 轮仍未修复**: 输出警告：

```
## ⚠️ 已修复 3 轮未解决

这个 bug 已经迭代 3 轮：
- 第 1 轮: <描述>
- 第 2 轮: <描述>
- 第 3 轮: <描述>

建议：
1. /end 归档当前进度，/clear 清洗上下文后重新审视
2. 或者描述更详细的复现步骤，我重新诊断
```

- **5 轮仍未修复**: 强制引导归档：

```
## 🛑 已修复 5 轮，强制归档

上下文可能已膨胀，修复质量下降。必须归档。

运行 /end 归档，然后 /clear 清洗上下文。
归档后用新会话重新审视这个问题。
```

## 注意

- **绝对禁止跳过诊断直接修复** — 这是漂移的根因
- 诊断时必须确认模块再动手，不确定就问用户
- 每次修复后必须提示用户重新测试，不要自己"测试"
- 轮次计数按模块+bug 组合去重，不同 bug 独立计数
- 3 轮警告、5 轮强制归档，不可跳过
