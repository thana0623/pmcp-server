# Tool Onboarding Guide — 工具引导上手系统

## Problem

用户下载并引入了开发工具（如 CodeGraph、ECC），但从未真正使用。工具停留在"已安装"状态，没有产出实际价值。

根本原因：
1. **不知道有什么能力** — 工具的命令、功能、适用场景不清晰
2. **不知道什么时候用** — 缺乏"这个场景应该用那个工具"的决策指引
3. **不知道效果如何** — 用完无法验证是否真的提升了效率
4. **表达偏差** — 想的是 A，表达的是 B，效果差导致大面积返工，最终放弃工具自己干

成本：工具装了不用 = 浪费磁盘 + 浪费决策精力 + 错失效率提升。

## Evidence

- CodeGraph：引入后使用 0 次，不知道是否安装成功、何时使用、效果如何
- ECC：内容体系庞大，用户理解太浅，触发效率低，有时直接跳过工具手动完成
- 观察到"装了不用"是反复出现的模式，不是个例

## Users

- **Primary**: 个人开发者（用户自己），搭建个人开发工作流，追求效率最大化
- **Not for**: 团队协作场景、CI/CD 自动化、非开发者用户

## Hypothesis

我们相信**在工具引入时自动生成引导流程（是什么 → 怎么用 → 何时用 → 验证效果）**会**解决"装了不会用、用了没效果"的问题**。

用了之后，**每个新工具在 5 分钟内能产出第一个实际成果，且用户能在真实开发场景中主动触发工具使用。**

## Success Metrics

| Metric | Target | How measured |
|---|---|---|
| 工具首次使用时间 | < 5 分钟（从引入到第一次有效使用） | 用户自报 |
| 工具主动触发率 | > 80% 场景下用户主动使用工具而非跳过 | 对话日志分析 |
| 返工率降低 | 表达偏差导致的返工减少 50% | 用户自评 |
| 工具留存率 | 引入的工具 7 天后仍在使用 | 用户自报 |

## Scope

**MVP — 三个核心能力**

1. **工具能力清单** — 引入工具后自动生成：有什么命令、解决什么问题、适用场景
2. **场景触发指引** — 在开发工作流各阶段，明确"此时应该用哪个工具"
3. **效果验证机制** — 使用工具后，提供验证方式确认工具是否生效

**Out of scope**

- 自动安装工具 — 用户自己负责安装
- 工具冲突解决 — 多工具冲突场景暂不处理，后续迭代
- 自动生成使用文档 — 不写教程，只做引导
- 自动配置工具 — 配置由用户手动完成

## Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| 工具种类太多，引导模板无法覆盖 | 高 | 中 | MVP 先覆盖 CLI 工具，后续扩展 API/库 |
| 用户忽略引导，和现在一样跳过 | 中 | 高 | 引导嵌入工作流关键节点，不是可选的 README |
| 表达偏差问题未解决 | 中 | 中 | 引导中加入"确认理解"步骤，强制对齐 |
